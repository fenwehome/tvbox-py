# 爱奇艺分类合并与筛选增强设计

日期：2026-05-06

## 背景

当前爱奇艺实现以 `pcw-api` 为主，首页分类与 `filters` 都是静态写死，适用于：

- 电影
- 剧集
- 纪录片
- 动漫
- 综艺
- 音乐
- 网络电影

这套实现的优点是稳定，但存在两个明显问题：

1. 分类范围偏窄，缺少用户要求的 `短剧`、`漫剧`、`少儿`、`知识`
2. 筛选能力受限于当前静态表，无法覆盖爱奇艺现网页面中的更完整分类和筛选能力

用户要求是“分类合并，筛选增强”，并明确选择优先完整而不是优先稳定。因此本次设计不再把所有类目硬绑定到同一套 `pcw-api` 分类接口，而是按爱奇艺现网页面结构拆分路由。

## 目标

- 在首页分类中新增 `短剧`、`漫剧`、`少儿`、`知识`
- 保留现有主类目的可用性
- 将分类与筛选能力从单一路由改为多路由
- 优先使用爱奇艺页面或频道的真实筛选结构
- 对页面未完整暴露的筛选项提供静态兜底
- 保持现有详情、搜索、播放链路尽量不受影响

## 非目标

- 不在本次中统一重构所有详情页逻辑
- 不追求所有类目都走同一接口
- 不为了统一编码保留现有 base64 复合筛选串方案
- 不在本次中覆盖爱奇艺所有内容频道，只覆盖本次目标类目

## 总体方案

采用“按频道族分路”的方案：

- `电影`、`剧集`、`纪录片`、`动漫`、`综艺`、`音乐`、`网络电影` 继续走 `pcw-api`
- `短剧`、`漫剧` 走专题或频道页抓取
- `少儿` 走儿童频道或儿童片库页抓取
- `知识` 走知识/教育频道抓取

这样做的核心原则是：

- 主类目优先复用当前稳定能力
- 新类目按真实页面结构接入
- 同一对外接口下允许多套内部数据来源并存

## 分类模型

首页分类扩展为：

- `电影`
- `剧集`
- `纪录片`
- `动漫`
- `综艺`
- `音乐`
- `网络电影`
- `短剧`
- `漫剧`
- `少儿`
- `知识`

`type_id` 改为显式路由编码，而不是单个纯数字。推荐格式如下：

- `pcw:1`
- `pcw:2`
- `pcw:3`
- `pcw:4`
- `pcw:5`
- `pcw:6`
- `pcw:16`
- `topic:short`
- `topic:comic`
- `child:kids`
- `edu:knowledge`

该编码直接表达“当前分类应该走哪条抓取链路”，避免在 `categoryContent()` 里不断堆叠特判。

## 路由设计

`categoryContent()` 不再直接承担所有逻辑，而是拆为三层：

### 1. route parser

输入 `type_id`，输出：

- `route_kind`
- `route_value`

示例：

- `pcw:2` -> `pcw`, `2`
- `topic:short` -> `topic`, `short`
- `child:kids` -> `child`, `kids`
- `edu:knowledge` -> `edu`, `knowledge`

### 2. filter builder

根据路由类型负责：

- 生成首页 `filters`
- 生成列表请求 URL 或请求参数
- 处理默认排序和默认筛选

这一层遵循“页面优先，静态兜底”：

- 优先读取页面现有筛选项
- 页面不完整时使用代码内稳定补全集合

### 3. list extractor

按不同数据源提取统一卡片字段：

- `vod_id`
- `vod_name`
- `vod_pic`
- `vod_year`
- `vod_remarks`

无论输入是 `pcw-api` JSON、频道 JSON 还是 HTML 页面，最终都收敛到同一输出结构。

## 筛选设计

筛选不再以“所有类目共用一个静态表”为目标，而是按路由分组。

### PCW 主类目

适用于：

- `电影`
- `剧集`
- `纪录片`
- `动漫`
- `综艺`
- `音乐`
- `网络电影`

筛选目标：

- 排序
- 年份
- 资费
- 地区
- 类型

实现策略：

- 保留当前稳定字段映射
- 能从页面补全的字段优先补全
- 无法稳定从页面解析的字段继续静态维护

### 短剧

筛选目标：

- 排序
- 资费
- 年份
- 类型

取舍：

- 不把地区作为首要筛选
- 重点支持免费/会员、上新、题材

### 漫剧

筛选目标：

- 排序
- 资费
- 年份
- 类型
- 地区

取舍：

- 视作更接近动漫/动态漫类目
- 保留地区和题材维度

### 少儿

筛选目标：

- 排序
- 年龄
- 类型
- 题材
- 资费

取舍：

- 年龄段优先级高于地区
- 如果页面能给出年龄和题材枚举，则直接使用页面值
- 如果页面无完整枚举，则静态提供常见年龄段和题材兜底

### 知识

筛选目标：

- 排序
- 领域
- 子类
- 资费

取舍：

- 用学科/知识方向替代传统影视的地区/年份优先模型
- 优先支持教育、科普、人文、历史、财经、科技等知识方向

## 筛选参数模型

现有实现包含 `_merge_filters()`，依赖将筛选值 base64 编码后再合并成复合串。该模型不适合多路由场景，原因如下：

- 新增类目不再共用一套参数语义
- HTML 列表页与 JSON 接口页参数结构差异大
- 继续堆叠复合编码会显著增加维护成本

本次设计改为普通键值对 `extend`：

- `{"mode": "11", "is_purchase": "0"}`
- `{"age": "3-6", "content_type": "儿歌"}`

每个路由处理器自己解释键值，不跨路由复用隐式编码逻辑。

## 列表抓取策略

### PCW 路由

保留当前 `pcw-api` 列表抓取能力：

- 继续使用现有接口
- 逐步补全筛选字段
- 保证现有主类目行为不回退

### topic 路由

用于：

- `短剧`
- `漫剧`

策略：

- 优先查找爱奇艺对应专题或频道页
- 优先解析页面中可复用的结构化数据
- 如果页面同时存在异步接口，则改抓接口，HTML 只做兜底

### child 路由

用于：

- `少儿`

策略：

- 优先使用儿童频道片库结构
- 列表页提取卡片时优先保留年龄/题材相关文案

### edu 路由

用于：

- `知识`

策略：

- 优先使用知识或教育频道页
- 允许该类目与传统影视类目完全不同的筛选结构

## 详情页兼容策略

本次不统一重构详情逻辑。

原则如下：

- 现有 `pcw` 类目详情继续复用当前逻辑
- 新增类目若能映射回现有 `albumId`、`pageUrl` 或 `playUrl`，则尽量复用现有详情能力
- 若个别新增类目只能拿到播放页 URL，则允许采用最小兼容详情策略

这可以把本次改动风险集中在“分类、筛选、列表”，避免一次性重构过多链路。

## 实施顺序

建议按以下顺序实施：

1. 保留现有 `pcw` 主类目可用
2. 引入新的分类定义与路由编码
3. 拆出 `route parser / filter builder / list extractor`
4. 接入 `短剧`、`漫剧`、`少儿`、`知识` 列表抓取
5. 补全主类目筛选能力
6. 对新增类目的详情做最小兼容

## 测试策略

测试按增量方式补齐。

### 首页分类测试

断言：

- 新增分类出现在 `homeContent()`
- `type_id` 使用复合路由编码
- 新增类目存在对应筛选定义

### 路由分发测试

断言：

- `pcw`
- `topic`
- `child`
- `edu`

都能正确分发到对应处理器。

### 列表解析测试

对每个新增类目准备 mock HTML 或 mock JSON：

- `短剧`
- `漫剧`
- `少儿`
- `知识`

断言能正确提取：

- `vod_id`
- `vod_name`
- `vod_pic`
- `vod_remarks`

### 筛选测试

断言：

- `少儿` 存在年龄筛选
- `知识` 存在领域筛选
- `短剧/漫剧` 存在排序和资费增强
- 主类目筛选仍兼容原有字段

## 风险与对策

### 风险 1：爱奇艺页面结构不统一

对策：

- 按路由拆分，不强行统一
- 优先解析结构化数据或异步接口
- HTML 解析只承担最后兜底

### 风险 2：新增类目详情不完全兼容

对策：

- 本次优先保证分类、筛选、列表
- 详情保持最小兼容，不做大重构

### 风险 3：筛选项来源不稳定

对策：

- 页面优先
- 静态兜底
- 对每条路由单独维护兜底表，而不是全局共表

## 成功标准

满足以下条件即视为本次目标达成：

- 首页可见 `短剧`、`漫剧`、`少儿`、`知识`
- 原有主类目可继续使用
- 新类目进入后能看到对应增强筛选
- 列表页能稳定返回可展示卡片
- 自动化测试覆盖分类、路由、筛选、列表解析的关键路径
